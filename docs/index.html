Receber Bank

Plataforma de recebimentos bancários baseada em microsserviços, eventos de domínio e separação entre escrita e leitura para emissão, consulta, pagamento e notificação de boletos.

Java 21 Spring Boot 3.2 PostgreSQL Redis Kafka / Redpanda Amazon EKS Amazon MSK DDD · CQRS · Outbox

Visão Geral

O projeto moderniza uma área de recebimentos bancários sem dependência de mainframe. A arquitetura usa serviços independentes por bounded context, eventos Kafka como bus de integração, o padrão Transactional Outbox para consistência banco/evento e um read model dedicado para consultas (CQRS).

Escrita transacional boleto-service persiste o boleto e grava o evento boleto.gerado na mesma transação ACID.
Eventos desacoplados Tópicos boleto.gerado, pagamento.efetivado e notificacao.enviada integram os serviços sem chamadas síncronas.
Consulta otimizada query-service mantém um read model próprio atualizado por eventos — nenhum serviço lê o banco do outro.
Pronto para cloud Infraestrutura AWS via Terraform e manifests Kubernetes prontos para EKS.

Padrões e Metodologias Aplicados

Padrão Onde se aplica no projeto
DDD — Bounded Contexts Cada microsserviço corresponde a um contexto delimitado: boleto, payment, query, notification. Entidades e regras vivem na camada domain, isoladas do framework.
Arquitetura Hexagonal Estrutura domain → application → adapter. Use cases dependem apenas de interfaces de porta; adapters (HTTP, Kafka, JPA) são plugáveis e substituíveis. Use cases testáveis sem Spring.
CQRS boleto-service é o modelo de escrita; query-service mantém seu read model próprio, atualizado por eventos Kafka. Nenhum serviço lê o banco de outro.
Transactional Outbox O boleto e o evento boleto.gerado são gravados na mesma transação PostgreSQL. O outbox-worker publica no MSK de forma independente — sem dual write, sem risco de inconsistência.
Event-Driven Architecture Amazon MSK (Kafka) desacopla todos os domínios. Serviços publicam e consomem eventos sem chamadas síncronas entre si.
Resiliência AWS SQS absorve picos na entrada (API Gateway → SQS → boleto-service). DLQ isola mensagens com falha. Kafka DLT por consumer com retry e backoff configurável via Spring Kafka.
Idempotência Idempotency-Key validada no ElastiCache Redis antes de processar cada mensagem — impede duplicatas em reprocessamentos do SQS ou Kafka.
Segurança em camadas CloudFront → AWS WAF → API Gateway (auth + throttling) → IAM/IRSA por service account no EKS → Secrets Manager para credenciais.

Decisões Arquiteturais (ADR)

As decisões mais relevantes estão documentadas com contexto, alternativas comparadas e consequências — úteis para explicar o porquê de cada escolha em revisões de código ou entrevistas técnicas.

ADR-001 · Transactional Outbox vs Saga Por que o Outbox venceu: uma única transação ACID cobre banco e evento, sem necessidade de coordenação distribuída ou framework de Saga para o escopo atual.
ADR-002 · CQRS — banco compartilhado vs separado Isolamento lógico em desenvolvimento, sem custo de dois clusters RDS. Em produção de alto volume, separar fisicamente é o próximo passo natural.
ADR-003 · Entrada assíncrona — API Gateway → SQS direto SQS absorve picos sem Lambda intermediário. A API retorna 202 Accepted e o serviço processa no seu ritmo, com DLQ isolando falhas.
ADR-004 · Arquitetura Hexagonal Use cases testáveis sem Spring, sem banco, sem Kafka. A troca de adapter (JPA, Kafka, in-memory) não afeta a regra de negócio.

Serviços

Serviço Porta Responsabilidade Tópicos
boleto-service 8081 Cria boletos, persiste no modelo de escrita e publica eventos via Transactional Outbox. Valida idempotência no Redis. Produz boleto.gerado
query-service 8082 Mantém o read model (CQRS) e expõe consultas por boleto ou status. Consome todos os eventos de domínio
payment-service 8083 Consome boleto.gerado, aguarda estímulo externo de pagamento e valida valor e vencimento. Consome boleto.gerado, produz pagamento.efetivado / pagamento.rejeitado
notification-service 8084 Gera notificações a partir de eventos de boleto e pagamento. Consome boleto.gerado e pagamento.*, produz notificacao.enviada

Fluxos das Jornadas

Fluxos separados para explicar cada parte do case sem misturar com a arquitetura geral.

1. Criação
Cliente envia POST /boletos; o boleto-service valida a requisição e cria o agregado de boleto.
2. Persistência e Outbox
O boleto é salvo no PostgreSQL e o evento boleto.gerado é gravado em outbox_events na mesma transação — garantia de consistência sem dual write.
3. Registro externo
A registradora bancária (CIP/Nuclea) é representada por uma API simulada em POST /simulacoes/registradora/boletos, deixando claro o limite com sistemas externos.
4. Aguardando pagamento
payment-service consome boleto.gerado apenas para saber quais boletos existem; ele não efetiva pagamento automaticamente.
5. Estímulo externo
O pagamento chega por POST /simulacoes/pagamentos, simulando retorno bancário, webhook, arquivo CNAB ou parceiro externo.
6. Validação
O pagamento é aprovado se o valor pago for igual ao valor do boleto e a data estiver dentro do vencimento. Caso contrário, publica pagamento.rejeitado.
7. Consulta e notificação
query-service atualiza o read model e notification-service notifica o resultado para o cliente PJ via notificacao.enviada.

APIs

boleto-service

MétodoEndpointDescrição
POST /boletos Cria um boleto e retorna 201 Created com Location apontando para o recurso.
GET /boletos/{id} Consulta um boleto pelo identificador no modelo de escrita.
POST /simulacoes/registradora/boletos Representa a API externa de registradora bancária / CIP / Nuclea para fins de demonstração.

query-service

MétodoEndpointDescrição
GET /consultas/boletos Lista boletos do read model. Aceita filtro opcional status.
GET /consultas/boletos/{boletoId} Consulta um boleto no read model pelo identificador.

payment-service — simulador

MétodoEndpointDescrição
POST /simulacoes/pagamentos Simula chegada de pagamento externo e publica pagamento.efetivado ou pagamento.rejeitado com base em valor e vencimento.

Ambiente Local

Use o Maven Wrapper para build e testes, e o Docker Compose para subir a stack completa.

cd receber-bank-services
./mvnw clean test
docker compose up --build

Dependências locais expostas:

PostgreSQL: 5432 Redis: 6379 Kafka / Redpanda: 9092

Endpoints de health check:

http://localhost:8081/actuator/health   # boleto-service
http://localhost:8082/actuator/health   # query-service
http://localhost:8083/actuator/health   # payment-service
http://localhost:8084/actuator/health   # notification-service

AWS e Kubernetes

O repositório contém base de infraestrutura em receber-bank-services/infra/aws/terraform e manifests Kubernetes em receber-bank-services/k8s/aws.

ÁreaRecursos provisionados
Rede e compute VPC, subnets públicas e privadas, NAT Gateway, EKS Cluster, node group, ALB Ingress Controller.
Dados e mensagens RDS PostgreSQL, ElastiCache Redis, SQS + DLQ, Amazon MSK, S3 para documentos e backups.
Segurança Security Groups, IAM roles com IRSA por service account, Secrets Manager e Kubernetes Secrets de exemplo.
Observabilidade CloudWatch Log Groups, métricas via Actuator / Prometheus, alertas SNS.

Para provisionar:

cd receber-bank-services/infra/aws/terraform
cp terraform.tfvars.example terraform.tfvars
# edite terraform.tfvars (obrigatório: db_password)
terraform init
terraform plan
terraform apply
